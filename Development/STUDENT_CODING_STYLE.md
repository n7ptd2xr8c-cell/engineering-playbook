# Student Coding Style

## Purpose

This document defines the preferred coding style used throughout my software engineering projects.

The objective is to produce code that is easy to understand, maintain, review, and improve while continuously adapting to my growing experience.

---

# General Philosophy

Readable code is more valuable than clever code.

A solution should first be:

- correct
- understandable
- maintainable

Only afterwards should optimization be considered.

---

# Adaptive Engineering

My coding style should evolve together with my engineering experience.

The quality of the code should always remain professional.

However, the implementation should match my demonstrated understanding rather than forcing advanced techniques unnecessarily.

As my skills improve, the coding style should naturally evolve.

---

# Simplicity

Prefer:

- simple logic
- small functions
- predictable behavior
- clear control flow

Avoid unnecessary abstractions.

If a simpler solution exists with the same quality, prefer the simpler one.

---

# Readability

Code should be easy to read from top to bottom.

Someone reading the file for the first time should quickly understand:

- what it does
- why it exists
- how it works

Avoid unnecessary nesting whenever possible.

---

# Naming

Choose meaningful names.

Variable names should describe the data they represent.

Function names should describe behavior.

Avoid abbreviations unless they are universally understood.

---

# Functions

Functions should:

- have one clear responsibility
- remain reasonably small
- receive only necessary parameters
- avoid hidden side effects

A function should be understandable without reading the entire project.

---

# Comments

Comments should explain:

- intent
- reasoning
- important decisions

Avoid comments that simply describe obvious code.

Good code should explain itself whenever possible.

---

# Error Handling

Handle errors explicitly.

Never ignore errors silently.

When an error occurs:

- understand why
- fix the root cause
- avoid temporary workarounds unless justified

---

# Debugging

Debug logically.

Read the error message.

Verify assumptions.

Understand the cause before changing code.

Avoid random experimentation.

---

# Code Reviews

During reviews:

- value constructive feedback
- prefer learning over ego
- improve code incrementally
- keep discussions objective

Every review should improve both the project and the developer.

---

# Continuous Improvement

Coding style is expected to evolve.

Each completed project should improve:

- readability
- engineering habits
- maintainability
- problem-solving ability

The objective is not to write perfect code.

The objective is to write better code than yesterday.
