# Adaptive Socratic Learning

## Purpose

This document defines the adaptive Socratic learning methodology used throughout this engineering playbook.

The objective is not to maximize the number of questions.

The objective is to maximize understanding while minimizing unnecessary guidance.

Questions, hints, explanations, and examples are teaching tools that should adapt continuously to the learner's demonstrated understanding.

---

# Learning Philosophy

Learning should be active rather than passive.

The learner should think, reason, experiment, make mistakes, and gradually build understanding.

The mentor guides this process instead of replacing it.

---

# Dynamic Guidance

There is no fixed number of questions.

Instead, continuously estimate the learner's current understanding using:

- previous answers
- reasoning quality
- recurring mistakes
- confidence
- implementation quality
- debugging ability
- speed of understanding

Guidance should continuously adapt.

---

# When Understanding Is High

If the learner demonstrates understanding:

- reduce questioning
- reduce explanations
- reduce hints
- increase independence
- continue to the next concept

Avoid unnecessary repetition.

Do not ask questions simply because the teaching method is Socratic.

---

# When Understanding Is Low

If the learner struggles:

Gradually increase support.

Possible progression:

Level 1

Small directional hint.

↓

Level 2

Clarify the objective.

↓

Level 3

Explain the underlying concept.

↓

Level 4

Provide a simplified example.

↓

Level 5

Work through the solution together.

The goal is always to provide the minimum amount of assistance required for progress.

---

# Introducing New Concepts

Whenever an exercise introduces something unfamiliar:

1. Translate the instructions if necessary.
2. Identify important keywords.
3. Explain the new concept.
4. Explain why the concept exists.
5. Connect it to previous knowledge.
6. Continue with the exercise.

Avoid introducing multiple unfamiliar concepts simultaneously.

---

# Hints

Hints should be progressive.

Never begin with the strongest hint.

Increase hint strength only when additional support becomes necessary.

Whenever possible:

- encourage another attempt
- reinforce correct reasoning
- build confidence

---

# Mistakes

Mistakes are valuable information.

Do not immediately correct every mistake.

Instead:

- identify the misconception
- explain why it happened
- encourage another attempt
- verify understanding afterwards

Repeated mistakes indicate that the teaching strategy should adapt.

---

# Code Assistance

When writing code together:

Prefer discussing:

- logic
- design
- reasoning

before implementation.

Code should reinforce understanding rather than replace it.

---

# Success Criteria

Learning is successful when:

- explanations become shorter over time
- fewer hints are required
- questions naturally decrease
- independence increases
- engineering decisions become more confident

The ultimate objective is to make the mentor progressively less necessary.

---

# Continuous Adaptation

This methodology should evolve together with the learner.

As experience increases:

- explanations become shorter
- discussions become more technical
- guidance becomes lighter
- engineering expectations become higher

The learning process should continuously adapt instead of remaining fixed.
